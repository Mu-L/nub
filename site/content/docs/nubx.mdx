Run whatever a name refers to with `nubx`: a file, a `package.json` script, a local bin from `node_modules/.bin`, or — on a local miss, and only with your consent — a package fetched from the registry. The local tiers cover the everyday case with no network and no prompt; the registry tier is the one nub guards.

```bash
nubx eslint .            # local bin from node_modules/.bin
nubx build              # package.json script "build"
nubx ./scripts/seed.ts  # a file
nubx cowsay "hi"        # not installed → ask, then fetch and run
```

For a locally-resolved bin, `nubx` does the `node_modules/.bin` walk-up in Rust and exec's the resolved binary directly, so the wrapper overhead that `npx` and `pnpm exec` pay on every call — a full Node bootstrap — approximately disappears: 19× lighter than `npx` on a native CLI like `esbuild` (and roughly 17× lighter than `pnpm exec`). The tool that runs is identical to what `npx` would run; only the wrapper gets out of the way.

## Resolution order

A bare token resolves by the first tier that matches, so `nubx <thing>` means the same thing as `<thing>` does to [`nub run`](/docs/run) and the file runner:

- **File** — a path (`./x`, `/abs/x`) or a file by that exact name in the current directory.
- **Script** — a `package.json` script of that name.
- **Bin** — an executable on the `node_modules/.bin` chain (workspace-hoisted bins included).
- **Registry** — a package fetched and run, when nothing local matched.

Scripts beat bins, and a file beats both — the same precedence as `run`. A local match never touches the network or prompts; only the registry tier does. A leading `-p`/`--package` spec is an explicit "fetch this package," so it skips local resolution and goes straight to the registry tier.

```bash
nubx tsc                 # the local typescript bin
nubx test                # the package.json "test" script, not a download
nubx ./build.ts          # a file, run through nub's TypeScript runtime
```

## Pass arguments

Everything after the name goes straight to the underlying file, script, or tool, untouched. There is no `--` separator to remember — `nubx` doesn't interpret the downstream flags.

```bash
nubx eslint . --fix --max-warnings 0
nubx tsc --noEmit
nubx vitest run --coverage
```

## Fetch from the registry

When nothing local matches, `nubx` can fetch the package, run its bin, and cache its consent — but it never executes remote code silently. The first time you run a given tool, `nubx` asks:

```console
$ nubx cowsay "Hello"
nubx: cowsay is not installed locally. Download and run it? [y/N] y
dependencies:
+ cowsay@1.6.0

 _______
< Hello >
 -------
        \   ^__^
         \  (oo)\_______
            (__)\       )\/\
                ||----w |
                ||     ||
```

Answer `y` once and nub records the consent: later runs of the same tool skip the prompt and run straight away. A pinned version (`nubx cowsay@1.5.0`) is remembered indefinitely; a floating one (`nubx cowsay`, which tracks `latest`) is re-confirmed after a day, so a tool that has moved to a new version asks again before running new code.

Outside an interactive terminal the answer can't be given, so `nubx` **fails closed** rather than guess — in CI, and any time stdin isn't a terminal:

```console
$ nubx cowsay "hi"          # in CI
nubx: refusing to download cowsay in CI.
  A CI job should declare the tool as a dependency, or pass -y to fetch it.
```

### `-y`

Pass `-y` (`--yes`) to consent up front: it lets a fetch through in CI or any non-interactive context, and skips the first-run prompt in a terminal. This is the deliberate "yes, download and run it" gesture for scripts and CI.

```bash
nubx -y cowsay "hi"
```

### `nub dlx`

When you want the explicit downloader with no resolution chain and no gate — invoking it *is* the consent — reach for `nub dlx` (and its short alias `nub x`, mirroring `bunx`/`bun x`). It fetches and runs a package directly, runs in CI without `-y`, and carries `-c`/`--shell-mode` and `--allow-build` for shell pipelines and lifecycle-script approvals.

```bash
nub dlx create-vite my-app
nub dlx -p cowsay -c 'cowsay hi | tr a-z A-Z'
```

`nub exec` is the opposite end: it runs a local bin and never reaches the registry at all — a missing bin there prints an install suggestion and exits 127.

## Coming from npx

`nubx <tool>` carries over, with two deliberate differences:

- **A local script or file wins, not just a bin.** `npx <name>` only ever resolves a bin (then downloads); `nubx <name>` resolves a file and a `package.json` script first. If a project has both a `lint` script and a `lint` bin, `nubx lint` runs the script — rename the script, or use `nub exec lint`, to force the bin.
- **A registry fetch needs consent.** Where `npx` downloads silently, `nubx` prompts on the first fetch and fails closed in CI / non-interactive contexts. A CI step that relied on `npx <tool>` auto-downloading must add `-y` or declare the tool as a dependency.

## `--node`

Pass `--node` to run the tool with runtime augmentation disabled — see [the runtime overview](/docs/runtime#--node) for the full contract.

```bash
nubx --node prisma generate
nub exec --node tsc --noEmit
```

What still happens under `--node`: the resolution chain, the registry fetch, and `nubx`'s own machinery. Only the runtime augmentation is off. Reach for it when a tool's shebang chain needs byte-exact plain Node, when bisecting a Nub-vs-Node issue, or when CI wants exact Node runtime behavior with `nubx`'s resolution.

## Supply-chain cooling window

A registry fetch runs through nub's normal install resolver, so the same `minimumReleaseAge` cooling window that protects `nub install` also protects a `nubx` fetch — a fetched tool's resolved version must be older than the window before it's used. The default is **24 hours**, the same as `install`; set `minimumReleaseAge=0` to disable it.

```console
# .npmrc: minimumReleaseAge tightened past every release,
# minimumReleaseAgeStrict=true
$ nubx -y cowsay "hi"
ERR_NUB_NO_MATURE_MATCHING_VERSION
  × dlx install failed
  ╰─▶ no version of cowsay matching latest is older than … minute(s)
      (minimumReleaseAgeStrict=true)
  help: blocked by age gate: 1.6.0
```

The window resolves from the same `.npmrc` scopes `npx` reads: the built-in default, your user-scope `~/.npmrc` (and `NPM_CONFIG_MINIMUM_RELEASE_AGE`), and a **project-local `.npmrc`** in the directory you run `nubx` from. So a per-project override applies even though the tool installs into a throwaway scratch project — alongside the project's registry, scoped registries, and auth, matching `npx`. By default the resolver falls back to the next-oldest satisfying version; `minimumReleaseAgeStrict=true` fails instead. See [the default-trust floor](/docs/install#default-trust-floor) for the full posture.

## Lifecycle scripts

A fetched package's `preinstall`/`install`/`postinstall` scripts stay **skipped** — the consent to download and run a tool is not consent to run its dependencies' build scripts. Approve a specific package's scripts with `--allow-build=<pkg>` on `nub dlx`, the same allowlist `nub install` uses.

## Yarn Plug'n'Play

Tools that only walk `node_modules/.bin/` — `npx`, `pnpm exec`, `bunx` — miss Yarn Berry's PnP bins. Nub handles them anyway: when the `.bin` walk-up misses in a PnP tree, `nubx` falls through to a PnP-aware runner that finds and executes the bin. See [module resolution](/docs/runtime/resolution#yarn-plugnplay) for how the shared PnP machinery works.
